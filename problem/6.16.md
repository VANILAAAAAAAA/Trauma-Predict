# EHR-Predict Input Design

## 1. Design goals

The patient's information is split into four views:

**STATIC** — admission-level context that does not change during the ICU stay.

**DAY** — long-history, coarse-granularity clinical summaries.

**HOUR** — short-history, fine-granularity recent physiology.

**CXR** — optional imaging-derived information, some of which is relevant to trauma.



Temporal structure of the input:
![{70AAB222-0D52-4D8F-BE6E-582A13FF0845}](C:\Users\a6439\Pictures\typora\{70AAB222-0D52-4D8F-BE6E-582A13FF0845}.png)

Starting from the prediction time `t`.

The current day is Day 0. Every day before Day 0 is cut into a complete 24-hour window. Day 0 runs from 00:00 to prediction time `t`, truncating at `t` when the elapsed time is less than 24 hours. This partial window, together with all prior complete windows, constitutes the coarse-granularity view.

From prediction time `t` looking back 24 hours, each hour is a window — 24 windows in total constitute the fine-granularity view.

Initially the two granularities were designed to be mutually exclusive: past complete ICU days use daily summaries, the most recent 24h use hourly resolution. However, the hourly view has limited carrying capacity — many other clinical signals cannot be expressed there. The incomplete Day 0 was therefore added as a coarse-granularity supplement. This means the current Day 0 and the most recent 24 hours of HOUR cover the same period, with one coarse and one fine resolution coexisting.



The four views are assembled into a single slot sequence and fed to a shared Transformer encoder.

![{EA09E75C-ADB9-4CCF-B194-DFFC312074C4}](C:\Users\a6439\Pictures\typora\{EA09E75C-ADB9-4CCF-B194-DFFC312074C4}.png)

Each part is processed differently.

**STATIC and DAY**: pure token input. A custom token vocabulary is defined for these two views, with a unique token tailored for each clinical situation. For DAY, which contains many fields and numeric representations, feature extraction and numeric bucketing are applied. No raw numeric values are used in STATIC or DAY — instead, each value is mapped to a range-based token. For example: age 20 maps to `[age_bin_18_39]`; age 41 maps to `[age_bin_40_54]`. The tokenizer and bucketing approach reference the hf_ehr codebase.

**HOUR**: pure numeric projection plus a ventilation status token. For the 7 vital signs, each variable at each hour is defined as one slot — 24 hours yields 168 slots total. Each vital sign passes through a simple MLP that maps the numeric value to an embedding of matching dimension, which is then combined with a time embedding. Ventilation is included as a separate token because it is a critical contextual variable that directly influences FiO₂ and RR within the vital signs.

**CXR**: under design. There is good news: MIMIC-CXR includes pre-computed CheXpert labels that can directly summarize CXR image findings. *CheXpert* is a standardized CXR dataset format that defines multiple finding labels, and its authors have released an open-source labeler model that can be reused directly. MIMIC provides the labeler's classification output — each image has a corresponding set of finding labels expressed as 0 or 1. These labels can be treated as text or designed as tokens and added to the current input, without the need to download the very large image dataset.

**Unifying the two outputs**: next hour head and report head both decode from the same latent state `z_t`. The next hour head performs 7-vital numeric regression; the report head performs event-level binary classification (1 = will occur, 0 = will not), ultimately taking the form of event prediction. Both heads share the encoder.

**Project roadmap**: V1 trains the next hour head first, with full input, aiming for the encoder to learn to compress the patient's history into a rich latent state `z_t`. V2 freezes the encoder and trains a report head on top of `z_t`.V3 jointly fine-tunes both heads.

## 3. DAY

### 3.1 Window definition

ICU day windows. Completed prior days are 24h full windows. `DAY_REL_0` is the current ICU day from midnight to prediction time `t`, truncated at `t`. Each DAY block always places a window-length token in the second position:

```text
[DAY_REL_-3] [day_window_len_24h] ...
[DAY_REL_0]  [day_window_len_13h] ...
```

When the elapsed window length is 0, `DAY_REL_0` is not emitted. To convey the duration of each window, every DAY-level window carries a length token — and for the final Day 0 window, 24 distinct tokens were designed, one for each possible length in hours. This lets the model understand how long each observation window covers.

DAY is a sparse summary. Numerical values are bucketized into clinical ranges rather than being passed as raw numbers, and feature extraction is applied. The clinical assessment is organized into 5 domains, each carrying highly condensed information. A token appears in a DAY window only when its event triggers — if nothing triggers, nothing is written, reducing redundant information. An additional data quality domain reports value coverage and observation rates.

### 3.2 Six domains

| Domain | Content |
|---|---|
| `[perfusion_shock]` | MAP low hours, SBP minimum, HR maximum, FIRST48 lactate/base deficit |
| `[oxygenation_ventilation]` | Ventilation hours, vent course, FiO₂ maximum, RR high burden |
| `[renal_metabolic]` | Creatinine change/ratio, bicarbonate minimum, BUN/Cr ratio, UOP (needs weight) |
| `[immune_hematologic]` | WBC high/low, RBC transfusion event |
| `[resuscitation_burden]` | RBC 48h, surgery 48h count, surgery in window, antibiotics in window, crystalloid (hold) |
| `[data_quality]` | Core vital coverage, lab draw status, UOP measurement status |

### 3.3 FIRST48

Lactate, base deficit, RBC 48h, and surgery 48h fields appear only once — on the second ICU day window (`source_day_index=1`) when the full first 48 hours are visible. They are not repeated on subsequent DAY blocks. If the history before prediction time `t` is shorter than 48 hours, these fields are not emitted to prevent leakage.

### 3.4 Variables per domain

| Domain | Variable |
| --------------------------- | ------------------------------------------------------------ |
| `[perfusion_shock]`         | `MAP` `SBP` `HR` `lactate`（FIRST48）`base_excess`（FIRST48） |

| `[immune_hematologic]`      | `WBC` `RBC`                                                  |
| `[resuscitation_burden]`    | `RBC`（48h）`surg48`（FIRST48）`surgSum` `antibiotics`       |

## 4. HOUR

<img src="C:\Users\a6439\Pictures\typora\{E656E3AE-90EA-4138-BE73-432A0D4C8652}.png" alt="{E656E3AE-90EA-4138-BE73-432A0D4C8652}" style="zoom:50%;" />

### 4.1 Engineering input

Not text. A fixed tensor:

```python
vital_values: [B, 24, 7]  # hr, sbp, dbp, map, rr, temp, fio2
vital_mask:   [B, 24, 7]  # 1=observed, 0=missing
```

Every hour has exactly 7 fixed slots. Missing values are represented by `mask=0`, not by deleting the slot. FiO₂ is the most sparse (MIMIC ~26%) but always occupies column 7.

### 4.2 Slot encoding

24×7 = 168 slots. Each slot is the sum of four components:

```text
slot = field_emb[f]        # which vital (7 options, embedding lookup)
     + hour_time_emb[t]    # which hour (24 options, embedding lookup)
     + mask_emb[obs]       # observed or missing (2 options, embedding lookup)
     + value_mlp(x_norm)   # standardized value → MLP(1→64), only when mask=1
```

The 168 slots are flattened into a `[168, D]` sequence and fed to the Transformer. `field_emb + hour_time_emb` jointly provide positional information for both cross-vital and temporal dimensions.

### 4.3 Physiological range filter

Filtering is mandatory before standardization to guard against extreme outliers — values outside range are set to `mask=0` (do not clip into range).

| Vital | Range |
|---|---|
| hr | 20–250 |
| sbp | 40–300 |
| dbp | 20–200 |
| map | 30–200 |
| rr | 4–80 |
| temp | 25–45 |
| fio2 | 0.21–1.0 |

### 4.4 Ventilation context token

Not placed inside the vital tensor. An independent sparse slot:

```text
slot = event_emb[vent_id]   # embedding lookup, no MLP
     + hour_time_emb[t]
```

Placed after the 7 vital slots of the same hour in the sequence. Its purpose is to tell the model that "this hour's vitals were measured under mechanical ventilation." This design follows the same event-trigger logic as DAY.

Bolus and RBC hourly events have been moved to DAY summary.

### HOUR head

Seven per-vital MLP heads read the current-hour corresponding slots:

```text
Transformer output [168, D]
  → z_hr   (hour 0, field 0) → MLP_hr   → HR prediction
  → z_sbp  (hour 0, field 1) → MLP_sbp  → SBP prediction
  → ...
```

### 4.6 Implementation

The MLP projection has been implemented and validated on a small sample.

##  CXR

Optional sparse event block. V1 uses pre-computed CheXpert labels from MIMIC-CXR (DenseNet-121 offline inference).

Only positive findings are emitted. 1760 of 6583 (26.7%) patients in C4 have an ICU-window CXR.

```text
[CXR_EVENT T=-52h]
[cxr_finding_cardiomegaly] [cxr_finding_edema]
[CXR_SEP]
```

Labels are embedded via lookup, identical to DAY tokens. When switching to VLM image embeddings in later versions, the interface remains unchanged.

## 6. Model

Shared Transformer encoder. Encoding path per slot type:

| Slot | Encoding |
|---|---|
| STATIC token | `token_emb[id]` |
| DAY token | `token_emb[id] + day_time_emb[rel] + segment_emb[DAY]` |
| DAY window length | `token_emb[id] + day_time_emb[rel]` |
| CXR time position | `time_emb[T_rel] + segment_emb[CXR]` |
| CXR finding | `token_emb[finding_id]` |
| HOUR vital | `field_emb[f] + hour_time_emb[t] + mask_emb[obs] + value_mlp(x)` |
| HOUR vent | `event_emb[vent_id] + hour_time_emb[t]` |

DAY and HOUR use independent time-embedding tables. `segment_emb` disambiguates time granularity.

Current dimension: `d_model=64`.

Typical slot budget:

```text
STATIC: ~12
DAY×4:  ~40
CXR×2:  ~6
HOUR:   168 + 0~24 vent
Total:  ~230
```

## 7. Full sequence

```text
[STATIC] ... [SEP]

[DAY_REL_-3] [day_window_len_24h] ... [SEP]
...
[DAY_REL_0] [day_window_len_13h] ... [SEP]

[CXR_EVENT T=-52h] ... [CXR_SEP]

[HOUR_REL_-23] ... [vent_on] [SEP]
...
[HOUR_REL_0] ... [CUR] [SEP]
```

`DAY_REL_0` and HOUR may overlap — DAY provides compressed same-day burden, HOUR provides high-resolution short-window detail. Both are truncated at the prediction point.

## 8. V1 scope

**V1 includes**: STATIC, DAY (with FIRST48, data quality, surgery, antibiotics), HOUR (168 vital slots + vent), CXR (CheXpert labels), shared encoder + 7 next-hour vital heads, physiological range filter, masked Huber loss.

**V1 excludes**: report generation, vasopressor/sedation tracking, crystalloid volume buckets, UOP KDIGO low-output (pending weight registry), NLR/strong ion, VLM image embedding.
